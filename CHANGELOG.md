# Changelog

All notable changes to the WCSAA Scoring System are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.0.0/).
Version scheme: **MAJOR.MINOR.PATCH**
- MAJOR — breaking change to data schema or scoring rules
- MINOR — new feature, new page, new trophy
- PATCH — bug fix, UI tweak, data correction

## [1.3.0] — 2026-06-01

### Added
- `supabase_client.py` — cached `@st.cache_resource` Supabase singleton (mirrors 4OAC pattern)
- `database.py` — full Supabase CRUD layer; all functions mirror the previous CSV API exactly
- `supabase_schema.sql` — run once in Supabase Dashboard to create all tables + indexes
- `migrate_csv_to_supabase.py` — one-time script to seed Supabase with existing CSV data
- `backup_supabase_data.py` — exports all Supabase tables to timestamped local CSVs
- `supabase>=2.3.0`, `Pillow>=10.0.0` added to `requirements.txt`

### Changed
- `app_lib.py` — all CSV read/write replaced with `database.py` calls; scoring, UI helpers, and logo management unchanged
- `theme.py` — `load_theme()` / `save_theme()` now persist to Supabase (`theme_config` table) with local file fallback
- `.streamlit/secrets.toml.example` — added `SUPABASE_URL` and `SUPABASE_KEY` entries

### Fixed
- **Data loss on Streamlit Cloud** — CSV files written via the UI were ephemeral (reset on every container restart). All app data is now persisted in Supabase PostgreSQL and survives restarts, re-deploys, and container recycling.
- **Concurrent write safety** — two simultaneous saves no longer risk CSV corruption.

### Database schema (Supabase tables)
| Table | PK | Notes |
|---|---|---|
| `seasons` | `season_id` | Active flag; CASCADE to all child tables |
| `anglers` | `(wp_no, season_id)` | Per-season roster |
| `competitions` | `(comp_id, season_id)` | Per-season schedule |
| `catches_raw` | `id` (serial) | Season-scoped raw entries |
| `catches_scored` | `id` (serial) | Re-generated by `rescore_all()` |
| `team_assignments` | `(comp_id, season_id, wp_no)` | Per-comp sub-team overrides |
| `trophy_nominees` | `(trophy, comp_id, season_id, wp_no)` | SDC nominees |
| `theme_config` | `id=1` | Singleton JSONB row |

## Branching & Release Workflow

```
main        ← production (Streamlit Cloud: wcsaa-scoring)
staging     ← staging    (Streamlit Cloud: wcsaa-scoring-staging)
feature/xxx ← feature branches (PR target: staging)
```

### To release a feature
1. `git checkout staging && git pull`
2. `git checkout -b feature/my-feature`
3. Make changes, commit, push
4. Open PR: `feature/my-feature` → `staging` on GitHub
5. Test on the staging Streamlit Cloud app
6. When satisfied, open PR: `staging` → `main`
7. Bump `__version__` in `version.py` and add entry to `CHANGELOG.md`
8. Merge to `main` — production auto-redeploys

### To hotfix production directly
1. `git checkout main && git pull`
2. `git checkout -b hotfix/description`
3. Fix, bump PATCH version, update CHANGELOG
4. PR: `hotfix/description` → `main`
5. After merge, backport: `git checkout staging && git merge main`
